



# 笔试题目标

## 一、背景与目标

我们正在寻找具备极强动手能力和 AI 工程化思维的伙伴。本次考核旨在通过构建一个 “智能招聘助手 MVP”，考察你如何利用前沿 AI 技术及现代 AI 工具链，解决业务场景中的实际痛点。

核心目标：

不要求实现商业级的全链路系统，但要求实现一个端到端闭环的核心功能模块。你需要展示从 “输入非结构化数据” 到 “输出结构化决策建议” 的完整 AI 处理流程。

------

## 二、任务描述

请搭建一个自动化的 AI 招聘演示系统（可 vibecoding）。为了降低难度并聚焦核心能力，我们建议你从以下两个场景中任选其一（也可两个场景串联）作为核心主线进行深度开发，其他环节可作为简化版或模拟接口存在。

### 场景 A：智能简历解析与试题生成引擎

- **输入**：上传一份 JD（职位描述）和若干份简历（PDF/Word）
- 核心功能
  - **结构化提取**：利用 AI 解析简历并提取关键信息，存入向量数据库或结构化存储。
  - **智能匹配打分**：基于业务需求与简历内容的匹配度，给出匹配度评分（0-100），并能给出精准的匹配度理由，决定是否推进正式面试。
  - **试题生成**：生成适合考察候选人的面试问题（至少 10 道），并给出试题的考察点、难度、评分标准等。
  - **追问模拟**：针对简历中的模糊点，自动生成 3-5 个面试追问问题。

### 场景 B：AI 模拟面试官 Agent

- **输入**：上传一份 JD（职位描述）和一份简历（PDF/Word）
- 核心功能
  - **角色设定**：根据上传的 JD，初始化一个特定风格的面试官 Agent（如：严厉的技术总监、亲切的 HR 等）
  - **多轮对话交互**：至少实现文字面试，若有余力可引入语音 TTS 或数字人。Agent 需具备记忆能力，能根据候选人的回答进行动态追问，而非机械问答。
  - **实时评估报告**：面试结束后，能自动生成一份面试评估报告，包含：岗位匹配度、沟通能力、潜在风险点及下一轮考察建议等。

## 三、技术要求与建议

我们不限定具体编程语言，但希望看到你对现代 AI 技术栈的熟练运用。

- 技术栈参考

  ：

  - LLM 接入：OpenAI API, Claude, Kimi, Qwen 等任意主流模型。
  - 框架 / 库：LangChain, LlamaIndex, Semantic Kernel, Flowise, Dify 或原生调用。
  - 数据存储：Vector DB (Chroma, Milvus, Pinecone) 或传统 DB (SQLite, PostgreSQL)。
  - 前端 / 交互：Streamlit, Gradio, Next.js, Vue 等，只要能直观演示交互即可。

- 工程规范

  ：

  - 代码需具备良好的可读性，关键逻辑需有注释。
  - 必须包含 `README.md`，详细说明环境配置、启动步骤及核心架构设计思路。
  - 敏感信息（如 API Key）请勿硬编码在代码中，请使用 `.env` 文件管理。

- **AI 编程辅助工具**：我们明确鼓励并允许你在开发过程中使用各类 AI 编程辅助工具来提升效率，包括但不限于 Claude Code、Codex、Qoder 等。

## 四、交付物清单

请在截止日期前提交一个 GitHub/GitLab 仓库链接，包含以下内容：

- 可运行的源代码：确保他人克隆后可通过简单命令启动 Demo。
- 演示视频：录制一段不少于 2 分钟的视频，完整演示系统从输入到输出的全过程。
- 设计文档（README）：
  - 架构图：简述数据流向和模块划分。
  - Prompt 设计思路：展示你如何优化 Prompt 以获得稳定输出（可选展示几个关键 Prompt）。
  - 难点与解决方案：你在开发过程中遇到的最大技术挑战及如何解决。

## 五、评估维度

我们将重点关注以下方面：

| 维度                | 权重 | 考察点                                                       |
| ------------------- | ---- | ------------------------------------------------------------ |
| AI 工程化能力       | 30%  | 是否合理使用了 Agent 编排和 Harness？Prompt 是否健壮？是否处理了幻觉或格式错误？ |
| 系统完整性          | 30%  | 核心闭环是否跑通？输入输出是否符合预期？异常处理是否得当？   |

| Demo 易用性与美观度 | 10%  | Demo 是否简单易用，是否美观大方等                            |
